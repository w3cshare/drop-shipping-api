# 三层补偿机制

## 概述

为确保 Shopify 订单数据的可靠同步，系统实现了三层补偿机制。当服务不可用或网络异常时，仍能保证订单数据不丢失。

## 架构图

```
+---------------------------------------------------------------+
|                      三层补偿机制                                |
+---------------------------------------------------------------+
|  第一层：Webhook 实时接收                                       |
|    Shopify Webhook -> WebhookController -> 写入 MySQL 队列       |
|    -> 立即返回 200                                              |
+---------------------------------------------------------------+
|  第二层：事件队列异步处理                                       |
|    WebhookEventProcessor -> 每 30 秒轮询 -> 处理订单/产品事件      |
|    -> 标记完成                                                  |
+---------------------------------------------------------------+
|  第三层：定时 REST API 全量同步                                 |
|    SyncScheduler -> 每 5 分钟 -> OrderSyncService -> REST API     |
|    -> 对比本地数据库 -> 补充缺失订单                              |
+---------------------------------------------------------------+
```

## 各层职责

### 第一层：Webhook 实时接收

**文件**：[webhook.controller.ts](file:///c:/Users/Admin/Desktop/com/shopify-project/growing-logistics-app/api-shopify/src/webhooks/webhook.controller.ts)

**职责**：
- 接收 Shopify 发送的 Webhook 事件（orders/create, orders/updated, products/create/update/delete）
- 立即将事件写入 MySQL 队列
- 立即返回 200 响应，避免 Shopify 重试

**优点**：
- 实时性最高，Shopify 创建订单后几秒内即可接收
- 立即写入数据库，即使服务重启也不会丢失

**代码示例**：
```typescript
@Post('orders/create')
@HttpCode(HttpStatus.OK)
async handleOrdersCreate(@Req() req: Request) {
  const order = req.body;
  const shop = req.headers['x-shopify-shop-domain'] as string;

  // 立即入队，立即返回 200
  await this.webhookQueueService.enqueue(shop, 'orders/create', order);

  return { success: true, queued: true };
}
```

### 第二层：事件队列异步处理

**文件**：
- 队列服务：[webhook-queue.service.ts](file:///c:/Users/Admin/Desktop/com/shopify-project/growing-logistics-app/api-shopify/src/webhooks/webhook-queue.service.ts)
- 事件处理器：[webhook-event-processor.ts](file:///c:/Users/Admin/Desktop/com/shopify-project/growing-logistics-app/api-shopify/src/webhooks/webhook-event-processor.ts)

**职责**：
- 存储 Webhook 事件到 MySQL 队列表
- 定时轮询待处理事件
- 异步处理事件（保存订单/商品到数据库）
- 失败自动重试（最多 3 次，间隔 1分钟/5分钟/15分钟）

**队列表结构**：`b_3rd_pending_events`

| 字段 | 说明 |
|------|------|
| id | 事件 ID |
| shop | 店铺域名 |
| event_type | 事件类型 |
| payload | 事件数据（JSON） |
| status | 状态：pending/processing/completed/failed |
| retry_count | 重试次数 |
| next_retry_time | 下次重试时间 |

**重试策略**：
- 第 1 次失败：1 分钟后重试
- 第 2 次失败：5 分钟后重试
- 第 3 次失败：15 分钟后重试
- 第 4 次失败：永久标记为 failed

### 第三层：定时 REST API 全量同步

**文件**：
- 同步服务：[order-sync.service.ts](file:///c:/Users/Admin/Desktop/com/shopify-project/growing-logistics-app/api-shopify/src/orders/order-sync.service.ts)
- 调度器：[sync-scheduler.ts](file:///c:/Users/Admin/Desktop/com/shopify-project/growing-logistics-app/api-shopify/src/orders/sync-scheduler.ts)

**职责**：
- 每 5 分钟执行一次增量同步
- 使用 Shopify REST API 拉取指定时间范围内的订单
- 与本地数据库对比，补充缺失的订单
- 记录同步状态，支持断点续传

**同步记录表**：`b_3rd_sync_records`

| 字段 | 说明 |
|------|------|
| shop | 店铺域名 |
| sync_type | 同步类型（orders/products） |
| last_sync_time | 最后同步时间 |
| last_sync_id | 最后同步的订单 ID（断点续传） |
| status | 同步状态：idle/syncing/error |

**同步策略**：
- **首次同步**：回溯最近 **7 天**（`INITIAL_SYNC_DAYS`）
- **增量同步**：回溯最近 **24 小时**（`SYNC_HOURS_LOOKBACK`）
- **智能检测**：如果 Shopify 订单数 > 本地数据库订单数，自动扩大到 7 天回溯范围
- **去重保证**：使用 `orderId + shop` 组合唯一约束 + TypeORM `upsert`

## 执行流程

### 正常情况

1. Shopify 创建订单 -> 发送 Webhook
2. WebhookController 接收 -> 写入队列 -> 返回 200
3. WebhookEventProcessor 轮询队列 -> 保存订单到数据库
4. 完成

### Webhook 队列积压情况

1. Shopify 创建订单 -> 发送 Webhook
2. WebhookController 接收 -> 写入队列 -> 返回 200
3. 服务重启，WebhookEventProcessor 重新启动
4. 启动时立即处理积压事件
5. 保存订单到数据库

### Webhook 完全不可用情况

1. 服务停机期间，Shopify 创建订单
2. 服务重启，WebhookEventProcessor 启动但队列为空
3. SyncScheduler 定时任务启动（onModuleInit 立即执行一次）
4. REST API 拉取最近 7 天的所有订单
5. 对比本地数据库，补充缺失的订单
6. upsert 操作保证不会产生重复

### 兜底同步机制（智能检测）

```
本地数据库: 2 条订单
Shopify: 4 条订单
   |
   v
首次同步：拉取 7 天内所有订单 -> 4 条
发现 Shopify(4) > 本地(2) -> 自动扩大到 7 天回溯
upsert 写入（不重复） -> 本地变为 4 条
```

## 定时任务配置

使用 `@nestjs/schedule` 装饰器管理：

| 任务 | 频率 | 装饰器 | 文件 |
|------|------|--------|------|
| 订单全量同步 | 5 分钟 | `@Cron('0 [slash]5 * * * *')` | sync-scheduler.ts |
| 队列状态检查 | 30 秒 | `@Interval('queueCheck', 30000)` | sync-scheduler.ts |
| 队列清理 | 1 小时 | `@Interval('cleanup', 3600000)` | sync-scheduler.ts |

> 注：实际 cron 字符串中 `[slash]` 表示斜杠分隔符的星号。

## 故障排查

### 队列中大量 pending 事件

**可能原因**：
- WebhookEventProcessor 未启动
- OrderService 保存订单失败

**排查方法**：
1. 检查服务日志，看是否有 `[WebhookEventProcessor] started` 日志
2. 查看 failed 事件的 last_error 字段
3. 查看队列统计：`GET /webhooks/queue/stats`

### 同步任务报错 "No access token found"

**可能原因**：
- 店铺未完成 OAuth 授权
- Session 已过期

**排查方法**：
1. 检查数据库 `b_3rd_shop_sessions` 表，确认 offline token 存在
2. 重新进行 OAuth 授权流程

### REST API 返回空数据

**可能原因**：
- 时间范围设置问题
- API 权限不足

**排查方法**：
1. 检查 `b_3rd_sync_records` 表的 `last_sync_time`
2. 确认 App 已申请 `read_orders` 权限
3. 首次同步回溯 7 天，避免遗漏历史订单

### 同步后出现重复订单

**根本原因**：缺少唯一约束

**解决方案**：
- 实体层已添加 `@Unique(['orderId', 'shop'])` 约束
- Service 层使用 `upsert` 替代 `findOne + save`
- 清理已有重复数据：
  ```sql
  DELETE FROM b_3rd_orders
  WHERE id NOT IN (
    SELECT MIN(id) FROM b_3rd_orders
    GROUP BY order_id, shop
  );
  ```
