## 1. 提交信息格式

```
<type>(<scope>): <subject>
```

- **`type`** – 必填，提交类别（仅限以下 7 种）。
- **`scope`** – 可选，影响范围（如模块、组件、页面等）。
- **`subject`** – 必填，简短描述，**不超过 50 个字符**。

> ⚠️ 注意：`()` 和 `:` 均为**英文半角**符号。

---

## 2. type 类型说明

| 类型 | 含义 |
|------|------|
| `feat` | 新功能（feature） |
| `fix` | 修补 bug |
| `docs` | 文档（documentation） |
| `style` | 格式（不影响代码运行的变动，如空格、缩进、CSS 等） |
| `refactor` | 重构（既不是新增功能，也不是修改 bug 的代码变动） |
| `test` | 增加测试（单元测试、集成测试等） |
| `chore` | 构建过程或辅助工具的变动（如 Webpack、依赖更新等） |

---

## 3. 规范细则

### 3.1 type 必须准确
- 每个提交只做一类事情，type 应与实际改动内容匹配。
- 不允许使用上述 7 种之外的任何标识。

### 3.2 scope 可选但建议明确
- scope 用于说明本次提交影响的**功能模块**、**组件**或**业务领域**。
- 若提交影响范围较广（如全局配置），可省略 scope。
- 示例：`设备列表`、`通用组件`、`登录模块`、`webpack` 等。

### 3.3 subject 简洁清晰
- 使用**祈使句**，采用中文或英文均可（团队统一）。
- 首字母小写（若用英文），结尾不加句号。
- 概括本次提交的核心变更，避免冗余。

---

## 4. 正确示例

### ✅ 带 scope 的提交
```
feat(设备列表): 新增设备列表静态页面
fix(设备列表): 修复请求参数缺失 name 字段
docs(readme): 增加构建参数说明
style(登陆): 修改登陆窗口背景为黑色
refactor(公共组件): 修改公共组件 chart-base 的实现逻辑
test(登陆): 增加登陆接口调用测试用例
chore(webpack): 添加自动打包插件 filemanager-webpack-plugin
```

### ✅ 不带 scope 的提交
```
feat: 新增设备列表静态页面
fix: 修复设备列表接口 bug
docs: 修改 readme 文档说明
style: 修改登录页样式
refactor: 重构公共组件
test: 添加测试脚本
chore: webpack 构建添加插件
```

### ✅ 特殊情况（描述 bug 序号）
```
fix: 修复 bug，序号为 1001
fix(bug[1001]): 修复请求参数缺失 name 字段
```

---

## 5. 错误示例（禁止）

```
feature: 新增页面          ❌ type 必须使用固定标识
fix(设备列表)修复bug       ❌ 缺少冒号和空格
fix: 修复了很多bug         ❌ subject 不具体
style: 修改代码格式，并修复样式 ❌ 混合多种类型
```

---

## 6. 自动化工具集成（可选）

- 可在 CI/CD 中增加提交信息校验（如 commitlint），强制符合本规范。
- 建议使用 `@commitlint/config-conventional` 配合自定义配置，仅允许上述 7 种 type。
